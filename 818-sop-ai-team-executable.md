# 818 AI 执行版 SOP
> 适用对象：AI团队（O/P/C角色的AI Agent）
> 版本：v1.0 | 2026-03-04
> 原则：能自动执行的直接执行；遇到 🔐 审批节点必须暂停，等待 Albert 确认后才能继续。

---

## 使用说明

每条SOP包含：
- **触发条件** — 什么情况下执行
- **执行步骤** — AI逐步操作
- **审批节点 🔐** — 必须发给 Albert 确认，不得跳过
- **输出格式** — 最终产物发给谁、长什么样

---

## ✅ 一类：AI完全自主执行

### AO-01 每日巡检报告
**触发条件**：每天 09:00 自动触发（Heartbeat）

**执行步骤**：
```
1. 拉取 Grafana 数据（或可访问的监控接口）：
   - 今日 QPS vs 昨日（计算波动率）
   - 错误率（目标<1%）
   - 今日营收
   - 活跃用户数

2. 检查 Telegram Bot 状态：
   - 发送 /start 测试是否在线
   - 检查未回复消息队列

3. 检查获客Agent：
   - 昨日采集线索数
   - 触达成功率
   - 账号是否有限制警告

4. 生成日报（见输出格式）并发送
```

**输出格式**：
```
📊 818 运营日报 YYYY-MM-DD

🔢 核心指标
  QPS：xxx（环比 +/-x%）
  错误率：x%（目标<1%）[正常/⚠️异常]
  今日营收：$xxx（环比 +/-x%）
  活跃用户：xxx

🤖 Bot状态：✅在线 / ❌异常
  未回复用户：x 条

📢 获客Agent
  昨日线索：xxx 条（目标>50）
  触达成功率：x%（目标>80%）
  账号状态：正常/⚠️有限制

⚠️ 异常记录
  [如无异常写"无"]

[AI-O 生成]
```
**发送目标**：研发+产品群

**触发条件**：每周一 09:00 自动触发

**执行步骤**：
```

  注册转化率：x%

⚠️ 本周问题
  [从日报汇总]

📌 下周计划
  [从迭代看板取Top3]

[AI-O 生成]
```

### AP-03 功能验收自动Checklist
**触发条件**：研发通知「xxx功能已部署到测试环境」

**执行步骤**：
```
1. 对照需求文档，逐项测试：

   功能验收：
   □ 主流程 Happy Path 跑通
   □ 异常场景：余额不足/网络超时/支付失败
   □ 俄语文案无错误
   □ Telegram Bot 显示正常

   数据验收：
   □ 监控埋点有数据上报
   □ 关键操作有日志记录

   安全验收：
   □ 用户只能查自己数据
   □ 金额无浮点异常

2. 生成验收报告发给 Albert
```

**审批节点 🔐**：
> 验收报告发给 Albert，等 Albert 回复「通过」后才能触发部署流程。

**输出格式**：
```
✅ 功能验收报告：[功能名]

功能验收：
  主流程：✅通过 / ❌失败（描述）
  异常场景：✅通过 / ❌失败
  俄语文案：✅正确 / ❌有误
  Bot显示：✅正常 / ❌异常

数据验收：
  埋点上报：✅ / ❌
  日志完整：✅ / ❌

安全验收：
  数据隔离：✅ / ❌
  金额计算：✅ / ❌

结论：[通过/不通过，不通过列出问题]

等待 Albert 审批上线 🔐
```

### AP-04 ROI 评估报告
**触发条件**：功能上线后第14天自动触发

**执行步骤**：
```
1. 从数据库/Grafana 拉取：
   - 功能使用次数/天（近14天均值）
   - 归因新增收入（近14天）
   - 开发成本（从需求文档取人天数）

2. 套用计算公式：
   ROI = (14天收入增量 / 开发成本折算) * 100%

3. 生成评估报告
```

**输出格式**：
```
📈 ROI 评估：[功能名]
上线日期：YYYY-MM-DD（已上线 14 天）

使用情况：x 次/天
收入增量：$xxx/周
开发成本：x 人天

ROI 结论：
  达预期 ✅ → 建议继续迭代
  未达预期 ⚠️ → 原因分析：[...]
  超出预期 🚀 → 建议加大投入

[AI-P 生成，供 Albert 决策参考]
```
**发送目标**：Albert

### AC-01 Git 规范检查
**触发条件**：收到 PR 请求 或 commit 消息

**执行步骤**：
```
1. 检查分支命名：
   □ 格式正确（feat/fix/hotfix/refactor + 描述）
   □ 从正确的父分支切出（feat→develop, hotfix→main）

2. 检查 commit message 格式：
   □ 符合 type(scope): description 格式
   □ type 为允许值（feat/fix/hotfix/refactor）

3. 不合规则自动评论，告知规范
```

**输出格式**：
```
🤖 Git规范检查

分支命名：✅合规 / ❌不合规（应改为：xxx）
Commit格式：✅合规 / ❌不合规（应改为：xxx）

[自动评论到 PR/Commit]
```

### AC-05 模型配置热更新（非定价变更）
**触发条件**：需要调整路由权重 或 开关某模型（非定价）

**执行步骤**：
```
1. 确认变更类型：
   - 定价调整？→ 🔐 停止，发给 Albert 审批
   - 纯路由/权重/开关？→ 继续

2. 备份当前配置：
   cp configs/model_config.yaml configs/model_config.yaml.bak

3. 修改配置文件

4. 热更新：
   POST /api/admin/reload-config

5. 验证生效：
   GET /v1/models 确认新配置

6. 监控 5 分钟（QPS/错误率无异常）

7. 记录变更日志并报告
```

**审批节点 🔐**：
> 若涉及任何定价字段变更 → 立即停止，发给 Albert 审批。

**输出格式**：
```
⚙️ 模型配置变更完成

变更内容：[描述]
执行时间：HH:MM
热更新状态：✅成功
5分钟监控：QPS正常 / 错误率正常

变更日志已记录。
[AI-C 执行]
```

### ACEO-04 供应商中断路由切换
**触发条件**：监控告警 某供应商错误率 > 20% 持续 5 分钟

**执行步骤**：
```
1. 确认故障供应商和影响范围：
   - 占总流量比例
   - 受影响模型列表

2. 立即调整路由权重（将故障供应商权重降为0）：
   修改 configs/model_config.yaml
   POST /api/admin/reload-config

3. 验证流量已切走（Grafana确认）

4. 发送报告给 Albert（信息通报，非审批）

5. 持续监控故障供应商，每 30 分钟更新一次状态
```

**审批节点 🔐**：
> 若需要调整用户定价补偿（例如降价促销）→ 发给 Albert 审批。

**输出格式**：
```
⚠️ 供应商中断自动处理报告

故障供应商：[名称]
发现时间：HH:MM
影响模型：[列表]
影响流量：x%

处理措施：
  已将流量切至：[备选供应商]
  路由切换时间：HH:MM
  当前错误率：x%（已恢复正常）

后续行动：
  持续监控故障供应商恢复情况
  每30分钟更新

[AI-C 自动处理，供 Albert 知悉]
```
**发送目标**：Albert + 研发群

### AO-02 账号池健康检查
**触发条件**：每日巡检发现账号状态异常 或 可用账号 < 3

**AI执行部分**：
```
1. 逐个测试账号（发送测试消息）
2. 分类：正常 / 受限 / 封号
3. 自动下架受限账号（从配置移除）
4. 生成账号状态报告
```

**审批节点 🔐**：
```
→ 需要购买新账号时：发给 Albert 审批（含数量+预估费用）
→ 冷却账号恢复使用前：发给 Albert 确认
```

### AO-03 USDT充值异常初查
**触发条件**：用户反馈充值未到账

**AI执行部分**：
```
1. 收集用户信息（金额/TxHash/时间）
2. 查询 TronScan 核实交易状态
3. 判断：
   - 确认数<3 → 告知用户等待
   - 地址错误 → 告知无法退款（标准话术）
   - 确认数≥3但未入账 → 生成工单发给 Albert
```

**审批节点 🔐**：
```
→ 需要手动入账：发给 Albert 确认（含TxHash+金额+用户ID）
→ 场景B重复入账：立即通报 Albert，暂停用户账户等待指令
```

### AC-04 生产事故协助响应
**触发条件**：错误率告警 > 5% 或 收到P0事故报告

**AI执行部分（止损前置）**：
```
1. 立即发P0告警到研发群
2. 拉取最近日志（docker logs 最近100行）
3. 对比 Grafana 异常时间点
4. 初步定位：是新部署引起？配置变更？外部依赖？
5. 生成初步诊断报告
```

**审批节点 🔐**：
```
→ 执行回滚前：必须发给 Albert 确认
→ 任何数据库写操作：必须 Albert 批准
→ 停止充值功能：必须 Albert 批准
```

### ACEO-05 危机公关起草
**触发条件**：服务宕机 > 30分钟 或 用户资金异常

**AI执行部分**：
```
1. 起草公告（中/俄双语）
2. 起草补偿方案建议
3. 发给 Albert 审批
```

**审批节点 🔐**：
```
→ 所有对外公告必须 Albert 审批后才能发布
→ 补偿金额必须 Albert 确认
→ AI不得独立发布任何危机声明
```

## 审批请求标准格式

当AI遇到审批节点，发给 Albert 的消息格式：

```
🔐 需要您的批准

SOP：[SOP编号]
操作：[具体要做什么]
影响：[会影响什么，金额/用户数/服务]
风险：[如不批准会怎样]

选项：
  A. 批准 — 继续执行
  B. 拒绝 — 终止流程
  C. 修改 — 告诉我怎么调整

等待您的回复。
[AI-{角色}]
```

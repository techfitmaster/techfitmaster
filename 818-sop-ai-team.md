> 适用范围：俄罗斯AI中转站 + 获客Agent

| 角色 | 英文 | 负责人 | 核心职责 |

## O — 运营 SOP

### O-01 每日巡检（每天09:00）

```
步骤：
1. 打开 Grafana 监控面板，检查：
   □ 网关 QPS（对比昨日，异常波动>30%需报告）
   □ 错误率（目标<1%，超过需立即通知研发）
   □ 今日营收（与昨日对比）
   □ 活跃用户数

2. 检查 Telegram Bot 状态：
   □ Bot 是否在线（发 /start 测试）
   □ 未回复用户消息（24h内未回复需人工跟进）

3. 检查获客Agent：
   □ 昨日采集线索数（目标>50条）
   □ 触达成功率（目标>80%）
   □ 账号是否有被限制警告

4. 记录到日报（Notion/飞书表格）
```

---

### O-02 获客Agent账号池管理

```
触发条件：账号被限制 / 低于3个可用账号

步骤：
1. 确认账号状态
   - 登录检测：尝试发送测试消息
   - 状态分类：正常 / 受限 / 封号

2. 受限账号处理
   - 立即下架该账号（从配置文件移除）
   - 启用备用账号
   - 受限账号冷却48小时后重试

3. 补充账号（账号<3个时）
   - 注册新Telegram账号（需俄罗斯手机号 或 虚拟号）
   - 新账号预热：先正常使用7天再加入触达池
   - 更新 config/account_pool.yaml

4. 记录到账号管理日志
   格式：{日期, 账号ID, 状态变更, 操作人}
```

---

### O-03 USDT充值异常处理

```
场景A：用户反馈充值未到账

步骤：
1. 要求用户提供：
   □ 充值金额
   □ 交易哈希（TxHash）
   □ 充值时间

2. 核查流程：
   - 去 TronScan 查询 TxHash
   - 确认：目标地址 / 金额 / 确认数（需≥3）
   
3. 处理：
   - 确认数<3：告知等待，一般10分钟内完成
   - 地址错误：无法退款，告知用户（提前在Bot写清楚）
   - 系统Bug未入账：手动触发入账，记录工单

场景B：重复入账

步骤：
1. 立即暂停该用户账户（防止套利）
2. 查询数据库确认重复次数
3. 手动扣除多余余额
4. 通知用户说明情况
5. 提交Bug给研发修复
```

### P-02 迭代规划（双周迭代）

```
迭代周期：2周一个版本

第1天（周一）：迭代启动
  - Albert 给出本迭代优先级列表
  - 研发评估工作量
  - 确认本迭代范围（不超过研发总容量80%）
  - 记录到迭代看板

第5天（周五）：中期同步（15分钟站会）
  - 已完成：x 个需求
  - 进行中：x 个需求
  - 风险：有无阻塞

第10天（周五）：迭代收尾
  - 完成验收测试
  - 上线部署
  - 记录迭代总结（完成率/未完成原因）
```

### P-03 功能验收 Checklist

```
上线前 Albert 逐项确认：

功能验收：
□ 主流程跑通（Happy Path）
□ 异常场景处理（余额不足/网络超时/支付失败）
□ 俄语文案无错误
□ 移动端显示正常（Telegram Bot）

数据验收：
□ 监控埋点正常（新功能有对应指标）
□ 日志记录完整

安全验收：
□ 用户只能看自己的数据
□ 金额计算无浮点精度问题

上线后30分钟：
□ 错误率未上升
□ 营收数据正常
□ 没有用户投诉
```

### P-04 ROI 评估（上线后2周）

```
对每个新功能做简单ROI评估：

指标收集（从Grafana/数据库）：
- 该功能使用率：x次/天
- 带来新增收入：$x/周
- 开发成本：x 人天

结论：
- 达预期 → 继续迭代
- 未达预期 → 分析原因，决定优化还是下线
- 超出预期 → 加大投入

记录格式：
{功能名, 上线日期, 2周使用率, 2周收入增量, 结论}
```

## C — 研发 SOP

### C-01 Git 分支规范

```
主分支：
  main        生产环境（受保护，不直接推送）
  develop     开发集成分支

工作分支（从develop切出）：
  feat/xxx    新功能
  fix/xxx     Bug修复
  hotfix/xxx  生产紧急修复（从main切出）
  refactor/xxx 重构

提交信息格式：
  feat(billing): 新增USDT充值低余额预警
  fix(bot): 修复/balance命令余额显示错误
  hotfix(gateway): 修复并发扣费负数问题

合并规则：
  - feat/fix → develop：需1人Code Review通过
  - develop → main：需Albert审批 + 通过CI
  - hotfix → main：Albert直接审批（紧急情况）
```

---

### C-02 代码审查（Code Review）规范

```
必须审查的内容：
□ 涉及金额/余额计算的代码
□ 数据库写操作（UPDATE/DELETE）
□ 新增对外API接口
□ 配置文件变更

审查标准：
□ 逻辑正确性（边界条件/异常处理）
□ SQL注入防护（参数化查询）
□ 敏感信息不进日志（API Key/USDT地址）
□ 关键操作有日志记录

时效要求：
- P0需求：2小时内完成Review
- 普通需求：24小时内完成Review
- 超时未Review：提交人可直接找人口头确认后合并
```

### C-03 上线部署 SOP

```
部署前（必做）：
□ develop 已通过CI（单元测试全绿）
□ 已在测试环境验证核心流程
□ 通知运营：「xxx功能将于HH:MM上线」
□ 确认回滚方案（镜像版本号记录）

部署步骤（Go服务示例）：
  1. docker build -t 818-ru-gateway:vX.Y.Z .
  2. docker push registry.aliyun.com/818/ru-gateway:vX.Y.Z
  3. 荷兰节点：docker pull + docker-compose up -d
  4. 检查健康：curl http://localhost:8080/health
  5. 观察5分钟错误日志

部署后（必做）：
□ 监控面板无异常（观察10分钟）
□ 通知运营：「xxx功能已上线，请验收」
□ 更新 CHANGELOG.md

回滚触发条件：
  错误率 > 5% 或 支付失败 或 收到用户投诉
  
回滚命令：
  docker-compose down
  docker pull 上个版本Tag
  docker-compose up -d
  预计恢复时间：5分钟内
```

### C-04 生产事故响应（P0级别）

```
定义：
  P0 = 服务完全不可用 / 支付资金异常 / 数据丢失

响应时效：
  P0：15分钟内响应，1小时内恢复或有临时方案

步骤：
1. 发现（0-5分钟）
   - 来源：监控告警 / 用户反馈 / 自查
   - 通知：立即发消息到研发群 「[P0] 服务名 - 现象描述」

2. 止损（5-15分钟）
   - 优先止损：回滚 / 降级 / 临时关闭问题功能
   - 不要在生产环境调试！先恢复再排查

3. 排查（15-60分钟）
   - 查日志：docker logs -f container_name
   - 查监控：Grafana异常时间点
   - 定位根因

4. 修复上线
   - 走hotfix分支流程
   - Albert快速审批

5. 事后复盘（24小时内）
   - 根因是什么？
   - 为什么没有提前发现？
   - 如何防止再次发生？
   - 更新本SOP或监控告警
```

### C-05 模型配置变更 SOP

```
场景：调整模型价格 / 开关某个模型 / 调整路由权重

变更前：
□ 评估影响：哪些用户受影响？预计收入变化？
□ Albert 审批（涉及定价必须审批）
□ 记录变更原因

变更步骤：
1. 修改 configs/model_config.yaml
2. 热更新（无需重启）：POST /api/admin/reload-config
3. 验证：GET /v1/models 确认新配置生效
4. 监控5分钟：调用量/错误率是否正常

变更后：
□ 记录变更日志：{时间, 变更内容, 操作人, 原因}
□ 如影响用户定价，Bot发通知告知用户
```

### C-06 数据库备份规范

```
自动备份（已配置定时任务）：
  - 频率：每天 02:00 全量备份
  - 保留：最近30天
  - 存储：阿里云OSS（异地）

手动备份触发条件：
  - 大版本上线前
  - 数据库结构变更前
  - 任何生产数据修改前

手动备份命令：
  mysqldump -u root -p 818_ru_db > backup_$(date +%Y%m%d_%H%M).sql
  ossutil cp backup_xxx.sql oss://818-backup/manual/

恢复流程：
  1. Albert 确认需要恢复的时间点
  2. 从OSS下载对应备份文件
  3. 测试环境先验证备份有效性
  4. 生产恢复（预计停服30分钟，提前通知用户）
  5. 恢复后数据验证
```

## OPC 协作规范

### 日常沟通节奏

```
每日：
  09:00  运营发日报（O）
  异步   需求/Bug 通过Issue提交（P→C）
  
每周：
  周一   迭代启动站会（OPC，30分钟）
  周五   迭代中期同步（OPC，15分钟）
  
双周：
  第2周周五   迭代验收 + 下迭代规划
```

### 紧急升级机制

```
普通问题  →  Issue / 群消息（24h响应）
P1问题    →  群消息@研发（2h响应）
P0问题    →  电话/微信直接联系（15min响应）

P0定义：支付异常 / 服务宕机 / 数据泄露
```

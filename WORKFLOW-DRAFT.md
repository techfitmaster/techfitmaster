```

    ↓

    ↓

```

## ⚙️ Phase 3: TDD 实现（我的职责）

### 我会做什么？

#### 3.1 Red 阶段（先写测试）
- 为每个 BDD 场景编写单元测试
- 运行测试 → **预期失败**（因为功能还未实现）

#### 3.2 Green 阶段（实现功能）
- **后端**：修改领域模型、应用服务、数据库脚本
- **小程序前端**：修改组件逻辑和 UI
- **管理后台**（如需要）：修改配置页面
- 运行测试 → **通过** ✅

#### 3.3 本地验证
- 执行 `./mvnw clean install -DskipTests`（编译验证）
- 执行核心测试验证功能正确性

### 我会通知你什么？
> "✅ [T3] TDD 实现完成，14 个测试全部通过"

## 🧹 Phase 4: Simplify + Refactor（我的自动化）

### 我会做什么？

#### 4.1 自动 Simplify（轻量清理）
- 删除未使用的导入
- 删除注释掉的代码
- 简化冗余逻辑
- 改进命名规范
- **强制编译验证**（确保没破坏功能）
- **自动运行测试**（确保简化后功能正常）

#### 4.2 Refactor 检查（智能建议）
检查以下重构机会：
- 重复代码（≥2次）→ 提取公共方法
- 长方法（>50行）→ 拆分
- 深层嵌套（>3层）→ Guard Clauses
- Platform Core 复用机会 → 提取到共享模块

**如果发现需要深度重构**：
> "检测到 3 处重复逻辑，建议提取公共方法。是否执行重构？"
> - 选项1: 「执行」→ 我自动重构
> - 选项2: 「跳过」→ 继续流程

### 我会通知你什么？
> "✅ 代码清理完成，删除 12 行冗余代码，编译和测试全部通过"

## 🚀 Phase 5: 推送与 CI 验证（自动化）

### 我会做什么？

1. **提交代码**：
   ```bash
   git add .
   git commit -m "feat(room): add SUPPORT role [T3]"
   git push origin feat-t3-support-role
   ```

2. **触发 GitHub Actions**：
   - **核心领域测试**（2-3分钟）：运行 domain.* 包的单元测试
   - **全量编译验证**（1-2分钟）：确保所有代码可编译

3. **监控 CI 状态**（每 30 秒刷新一次）

### CI 通过 ✅
> "🎉 [T3] CI 验证通过！正在创建 PR..."

### CI 失败 ❌
**我的自愈流程**：
1. 自动下载失败日志
2. 分析根因：
   - 编译错误 → 修复语法/依赖问题
   - 测试失败 → 修复测试逻辑或业务代码
   - 红线违规（如 `.last()` 拼接 SQL）→ 按规范修复
3. 自动修复代码并重新推送
4. 重复直到 CI 变绿 ✅

**我会实时同步进度**：
> "⚠️ CI 失败：测试 TC-UT-ROOM-101 断言错误"  
> "🔧 正在自动修复..."  
> "✅ 已修复并重新推送，等待 CI 重新验证..."

## 📊 质量保障机制

### P0 红线检查（违反会阻塞合并）

| 维度 | 检查规则 | 检测方式 |
|------|---------|---------|
| **安全性** | 禁用 `.last()` 拼接 SQL | CI 自动扫描 |
| **安全性** | 资金必须用 BigDecimal | CI 自动扫描 |
| **稳定性** | 余额操作必须加分布式锁 | CI 自动扫描 |
| **复用性** | 禁止重复实现 platform-core 组件 | 我在开发时检查 |
| **规范** | 实体类禁用 @Data | CI 自动扫描 |

### 测试覆盖要求

| 场景类型 | 最低数量 |
|---------|---------|
| Happy Path（正常流程） | 3+ |
| Exception Path（异常处理） | 2+ |
| Boundary（边界条件） | 1+ |

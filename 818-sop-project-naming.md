# 818科技 项目定义与命名 SOP
> CTO 制定 | v1.0 | 2026-03-04
> 适用范围：所有新建工程、模块、服务

---

## 一、核心原则

1. **名称即定位** — 看到名字，立刻知道这个项目做什么
2. **前缀统一** — 所有项目以 `818` 开头（公司标识）
3. **独立即独立** — 独立部署 = 独立仓库，不放在同一 repo 下混搭
4. **定义先于开工** — 项目开始前必须完成本 SOP 的命名检查清单

---

## 二、命名规范

### 格式
```
818-{领域}-{职责}
```

| 字段 | 说明 | 示例 |
|------|------|------|
| `818` | 公司前缀，固定 | 818 |
| `{领域}` | 业务领域，1-2个英文单词 | relay / scout / proxy / growth |
| `{职责}` | 可选，进一步区分同领域多服务 | （留空或加后缀） |

### 合法示例
```
818-relay        ✅ 转化追踪代理服务
818-scout        ✅ Telegram获客Agent
818-proxy        ✅ AI中转站网关
818-growth       ✅ 获客系统（未来多渠道整合）
818-console      ✅ 管理后台
818-pay          ✅ 支付模块（若独立）
```

### 禁止示例
```
818platform      ❌ 缺少连字符，不清晰
relay            ❌ 无818前缀
818-platform     ❌ "platform"太泛，不表达职责
818_relay        ❌ 下划线，统一用连字符
Relay818         ❌ 前缀不在开头
```

## 三、项目分类定义

### 3.1 独立服务（Independent Service）
- **定义**：独立部署、独立进程、有自己的端口/容器
- **要求**：独立 Git 仓库 + 独立 go.mod（或 package.json）
- **命名**：`818-{领域}`
- **示例**：818-relay、818-scout、818-proxy

### 3.2 内部模块（Internal Module）
- **定义**：不独立部署，作为某个服务的内部子包
- **要求**：放在所属服务的 `internal/` 目录下，不建独立 repo
- **命名**：`internal/{包名}`（无818前缀）
- **示例**：`818-scout/internal/scorer`、`818-relay/internal/handler`

### 3.3 工具脚本（Utility Script）
- **定义**：运维脚本、迁移脚本、一次性工具
- **要求**：放在所属服务的 `scripts/` 目录下，或 workspace 根目录
- **命名**：动词描述，如 `migrate-db.sh`、`seed-data.go`

## 四、Go Module 命名规范

```
module github.com/818tech/{项目名}
```

示例：
```go
// 818-relay/go.mod
module github.com/818tech/818-relay

// 818-scout/go.mod
module github.com/818tech/818-scout
```

---

## 五、Git 仓库命名规范

- **GitHub 组织**：`818-tech`（对外）或 `techfitmaster`（现有）
- **仓库名 = 项目名**：`818-relay`、`818-scout`
- **默认分支**：`main`
- **仓库可见性**：默认 Private

## 六、开工前检查清单（必须完成）

每个新项目开工前，Albert 确认以下事项：

```
项目名：___________
确认日期：___________

□ 1. 命名符合 818-{领域} 格式
□ 2. 领域定义清晰（一句话描述职责）
□ 3. 确认是独立服务还是内部模块
□ 4. 架构文档已存在（ARCH-CONFIRMED.md 或等效文件）
□ 5. 已检查 SOP 合规性（功能覆盖范围与 SOP 要求对齐）
□ 6. 数据库表设计已确认（如涉及DB）
□ 7. 对外接口已定义（API 列表）
□ 8. MVP 交付范围已明确（不超出范围开工）
```

## 八、命名变更流程

已上线项目的命名变更成本极高（影响域名、CI/CD、文档），原则上**不允许**。

若确实需要变更：
1. Albert 确认变更必要性
2. 提前 2 周通知所有相关方
3. 新旧名称并行运行 1 周（redirect/alias）
4. 完成后更新本 SOP 命名表
